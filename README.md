# Зерде Керамика Актобе — CRM для керамогранитного завода

Production-ready CRM для управления полным циклом сделки керамогранитного завода:
от приёма заявки клиента до закрытия сделки после доставки.

- **Frontend:** React + TypeScript, Vite, Tailwind CSS, dnd-kit (канбан), lucide-react, TanStack Query, Recharts, Socket.IO.
- **Backend:** Node.js + Express + TypeScript, Prisma ORM, JWT-авторизация, Socket.IO (realtime), Multer (файлы), Zod (валидация), RBAC на сервере.
- **БД:** PostgreSQL (production). Для локального запуска по умолчанию используется SQLite на той же Prisma-схеме (см. ниже) — переключение на Postgres в 2 строки.

Язык интерфейса — русский (вынесен в словари, готов к EN/KZ). Валюта и бренд настраиваются.

## Быстрый старт

Требуется только **Node.js 18+** (проверено на Node 24). Docker/PostgreSQL не нужны для локального запуска.

```bash
# 1. Установить зависимости (сервер + клиент)
npm run install:all

# 2. Создать БД (SQLite), применить схему и залить демо-данные
npm run db:setup

# 3. Запустить backend (:4000) и frontend (:5173) одновременно
npm run dev
```

Откройте **http://localhost:5173** и войдите под одной из тестовых учёток ниже.

> Если порт 4000/5173 занят — поменяйте `PORT` в `server/.env` и `VITE_API_URL` в `client/.env`.

## Переключение на PostgreSQL

Схема Prisma написана так, что работает и на SQLite, и на PostgreSQL без изменений
(не используются native enum/array — статусы хранятся строками с валидацией в приложении).

1. Поднимите Postgres (любой способ):
   - **Docker:** `docker compose up -d db` (см. `docker-compose.yml`)
   - **Облако:** бесплатный Postgres на [Neon](https://neon.tech) или [Supabase](https://supabase.com)
2. В `server/.env` поменяйте:
   ```dotenv
   DATABASE_PROVIDER="postgresql"
   DATABASE_URL="postgresql://crm:crm@localhost:5432/crm?schema=public"
   ```
3. В `server/prisma/schema.prisma` смените `provider = "sqlite"` → `provider = "postgresql"`.
4. Примените схему и сид:
   ```bash
   npm run db:setup
   ```

Для production используйте `prisma migrate deploy` (миграции в `server/prisma/migrations`).

## Роли и права (RBAC)

Права проверяются **на сервере** (middleware `requireRole` + проверка владения объектом),
не только скрытием кнопок в UI.

- **Клиент** — создаёт заявки, видит только свои, отслеживает статус (трекер этапов + история),
  загружает акт выполненных работ, подаёт рекламацию. Не видит чужие заявки, закупочные цены, внутренние комментарии.
- **Менеджер** — ведёт сделку по всему циклу: дебиторка, спецификация/договор, условия оплаты,
  ТТН/УПД, контроль документов, закрытие. Видит заявки своих клиентов.
- **Завод** — канбан заявок в производстве, план производства, приоритеты, статусы
  «запущено → готово → передано на склад».
- **Склад** — наличие, резерв под заявку, частичная/полная отгрузка, передача остатка в производство.
- **Логист** — заявки, готовые к отгрузке; маршрут и доставка; «товар передан клиенту».
- **Бухгалтер** — дебиторка, статусы оплат, счета, подтверждение оплаты, одобрение постоплаты, разблокировка по долгу.
- **Администратор** — пользователи, роли, справочники (заводы, перевозчики, номенклатура), настройки.

## Бизнес-процесс (этапы сделки)

Заявка (`Order`) проходит статусы; канбан = колонки-этапы, карточка двигается по циклу.
Каждый переход логируется в `OrderHistory` (кто, когда, с какого → на какой).

```
Новая заявка → Проверка дебиторки ─(долг)→ ОТКЛОНЕНО (терминальный)
                                 └(нет долга)→ Согласование (спецификация)
   → На подписании (подписи менеджера и клиента)
   → [Аванс] Ждём оплату  /  [Постоплата] одобрение
   → Документы подтверждены → Резервирование на складе
   → Наличие? ─полное→ сразу Отгрузка
              ├частично→ частичная отгрузка + остаток в производство
              └нет→ Производство
   → Производство (правило 15-го числа; Приоритет 1 аванс / 2 постоплата) → Готово → на склад
   → Отгрузка (ТТН + УПД) → Доставка (товар передан) → Ожидание документов (акт)
   → Претензия? ─да→ Рекламация → возврат в цикл
   → [Постоплата] счёт + оплата
   → СДЕЛКА ЗАКРЫТА
```

**Прямой путь:** товар на складе полностью → производство минуется (склад → отгрузка → доставка).

## Ключевые бизнес-правила

Реализованы в `server/src/domain` и сервисах:

1. **Проверка дебиторки** блокирует движение: есть долг → авто-перевод в «Отклонено» с причиной;
   разблокировка бухгалтером.
2. **Правило 15-го числа:** заявка до 15-го → план на следующий месяц; после 15-го → на послеследующий.
3. **Приоритезация:** аванс → Приоритет 1; постоплата → Приоритет 2 (после оплаченных).
4. **Частичная отгрузка:** не хватает товара → отгружаем наличие, остаток автоматически в план производства.
5. **Авто-напоминания** менеджеру, если акт не получен в срок.
6. **Аудит:** каждое изменение статуса логируется.
7. **Уведомления по ролям:** клиенту — смена статуса; менеджеру — новая заявка/подпись/просрочка; заводу — новые позиции плана.

## Безопасность и нефункциональные требования

- Проверка прав на сервере (RBAC middleware + ownership), валидация ввода (Zod), защита файлов по ролям.
- Пагинация списков и заявок; канбан группируется по статусам.
- Локализация вынесена в словари (`client/src/lib/i18n.ts`), готовность RU/EN/KZ.
- Демо-данные: клиенты, ~18 заявок на разных этапах, номенклатура керамогранита, остатки, план производства.

> **Демо-упрощения:** денежные поля — `Float` (для production → `numeric`); файловое
> хранилище по умолчанию локальное (`STORAGE_DRIVER=local`), для production — S3 (MinIO в docker-compose).
